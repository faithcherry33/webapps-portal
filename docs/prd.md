# 학습용 웹앱 포털 PRD

문서 버전: v0.2  
수정 핵심: 관리자 코드 기반 앱 추가 기능, 목적별 카테고리, 태그 기반 정렬/필터링 반영

---

## 1. 제품 개요

### 1.1 제품명
가칭: 우리반 학습 포털

### 1.2 한 줄 설명
선생님이 만든 학습용 웹앱을 한곳에 모아두고, 학생이 카테고리와 태그를 통해 필요한 앱을 쉽게 찾아 실행할 수 있는 앱 런처형 웹 포털.

### 1.3 만들려는 이유
여러 학습용 웹앱이 각각 다른 주소로 흩어져 있으면 학생은 어떤 앱을 눌러야 하는지 헷갈리고, 교사는 수업마다 링크를 따로 안내해야 한다.  
웹 포털은 모든 웹앱을 한 화면에 모아 보여주고, 교사가 직접 앱 링크와 썸네일을 등록해 계속 확장할 수 있게 한다.

### 1.4 핵심 방향
- 학생 화면은 앱 런처처럼 단순하고 직관적으로 만든다.
- 앱 분류는 과목이 아니라 사용 목적 중심으로 나눈다.
- 세부 주제는 `#태그`로 관리한다.
- 교사는 관리자 코드를 입력한 뒤 새 웹앱을 직접 등록할 수 있다.
- 1차 버전부터 앱 추가 기능을 포함하되, 복잡한 회원 관리 기능은 넣지 않는다.

## 2. 대상 사용자

### 2.1 주요 사용자
초등학생

### 2.2 보조 사용자
교사

### 2.3 사용자별 목표

#### 학생
- 포털에서 필요한 학습 앱을 빠르게 찾는다.
- 전체, 수업, 학습, 놀이, 학급운영 카테고리로 앱을 살펴본다.
- `#독서`, `#경제`, `#빙고`, `#퀴즈` 같은 태그를 눌러 비슷한 앱을 모아 본다.
- 앱 카드를 눌러 바로 활동을 시작한다.

#### 교사
- 관리자 코드를 입력하고 앱 추가 화면에 들어간다.
- 앱 제목, 링크, 카테고리, 태그, 썸네일 이미지를 직접 등록한다.
- 새 앱을 추가하면 학생용 포털에 바로 반영되게 한다.
- 앱 목록을 과목명이 아니라 수업 목적과 활동 성격 중심으로 관리한다.

## 5. 카테고리 정책

### 5.1 카테고리 목록
카테고리는 과목별로 만들지 않고, 앱을 사용하는 목적에 따라 나눈다.

- 전체
- 수업
- 학습
- 놀이
- 학급운영

### 5.2 카테고리 의미

#### 전체
모든 공개 앱을 보여주는 기본 보기이다.  
데이터에는 저장하지 않고 화면 필터로만 사용한다.

#### 수업
교사가 수업 시간에 직접 안내하여 사용하는 앱이다.  
예: 독서 퀴즈, 도덕 토론, 사회 개념 활동, 수업 중 실시간 활동

#### 학습
학생이 개념을 익히거나 복습하는 데 쓰는 앱이다.  
예: 개념 복습, 단어 연습, 문제 풀이, 자기주도 학습

#### 놀이
게임형, 빙고형, 워드서치형 등 놀이 요소가 강한 앱이다.  
예: 개념 빙고, 낱말 찾기, 퀴즈 게임, 미션 활동

#### 학급운영
수업 외에 학급 생활, 학급 규칙, 의견 수렴, 역할 정하기 등에 쓰는 앱이다.  
예: 학급 회의, 자리 바꾸기, 투표, 생활 약속, 칭찬 게시판

### 5.3 카테고리 선택 규칙
- 앱 하나는 기본적으로 하나의 대표 카테고리를 가진다.
- 여러 성격을 동시에 가진 앱은 대표 카테고리 하나를 선택하고, 나머지 성격은 태그로 표현한다.
- 예: 독서 퀴즈 앱은 카테고리를 `수업`으로 두고, 태그를 `#독서`, `#퀴즈`, `#문해력`으로 설정한다.

## 6. 태그 정책

### 6.1 태그의 역할
태그는 앱을 세부 주제나 활동 방식별로 묶는 데 사용한다.

예:
- #독서
- #질문
- #퀴즈
- #경제
- #사회
- #도덕
- #빙고
- #워드서치
- #토론
- #복습
- #문해력
- #학급회의
- #투표
- #생활지도

### 6.2 태그 입력 규칙
- 교사는 앱 등록 시 태그를 직접 입력할 수 있다.
- 태그는 쉼표 또는 Enter로 구분해 여러 개 입력할 수 있다.
- 사용자가 `#`을 입력하지 않아도 시스템이 자동으로 `#` 형태로 표시한다.
- 같은 앱 안에서 중복 태그는 저장하지 않는다.
- 태그는 공백 없이 짧게 입력하는 것을 권장한다.
- 예: `독서, 퀴즈, 문해력`

### 6.3 태그 필터 규칙
- 학생이 태그를 누르면 같은 태그가 달린 앱만 모아 보여준다.
- 카테고리와 태그 필터는 동시에 적용할 수 있다.
- 예: `수업` 카테고리에서 `#독서`를 누르면 수업 카테고리 안의 독서 관련 앱만 표시한다.
- 선택된 태그는 화면에서 강조 표시한다.
- 태그 선택을 해제하면 원래 목록으로 돌아간다.

### 6.4 태그 정렬 규칙
- 태그 목록은 기본적으로 사용 빈도가 높은 순서로 보여준다.
- 같은 빈도라면 가나다순으로 정렬한다.
- 앱 카드 안의 태그는 등록 순서대로 보여준다.
- 추후 사용량 데이터가 생기면 자주 실행된 태그를 우선 표시할 수 있다.

### 7.3 관리자 코드 입력 화면

#### 목적
교사만 앱 추가 화면에 들어갈 수 있게 한다.

#### 화면 구성
```text

[코드 입력창]

```

#### 동작
- 올바른 관리자 코드를 입력하면 관리자 모드로 이동한다.
- 틀린 코드를 입력하면 오류 문구를 표시한다.
- 관리자 인증 상태는 브라우저 세션 동안 유지할 수 있다.

#### 오류 문구 예시
```text

다시 입력해 주세요.
```

## 8. 기능 요구사항

### 8.1 앱 목록 표시
- 시스템은 공개 상태인 앱만 학생 화면에 표시해야 한다.
- 앱은 기본적으로 최신 등록순 또는 정렬 순서순으로 표시한다.
- 앱 카드에는 제목, 썸네일, 설명, 카테고리, 태그, 시작하기 버튼을 표시한다.

### 8.2 카테고리 필터
- 사용자는 전체, 수업, 학습, 놀이, 학급운영 카테고리를 선택할 수 있어야 한다.
- 전체를 선택하면 모든 공개 앱을 보여준다.
- 특정 카테고리를 선택하면 해당 카테고리 앱만 보여준다.
- 선택된 카테고리는 시각적으로 강조한다.

### 8.3 태그 필터
- 앱 카드의 태그를 누르면 같은 태그가 있는 앱만 보여준다.
- 상단의 자주 쓰는 태그를 눌러도 같은 방식으로 필터링한다.
- 선택된 태그는 강조 표시한다.
- 태그 필터 해제 버튼을 제공한다.

### 8.4 검색
- 사용자는 검색창에 앱 제목, 설명, 태그를 입력해 앱을 찾을 수 있어야 한다.
- `#독서`처럼 태그 형태로 검색해도 결과가 나와야 한다.
- 검색어와 카테고리, 태그 필터는 동시에 적용될 수 있다.
- 검색 결과가 없으면 안내 문구를 보여준다.

### 8.5 앱 실행
- 시작하기 버튼을 누르면 앱 링크로 이동한다.
- 앱 링크는 기본적으로 새 탭에서 열린다.
- URL 형식이 올바르지 않으면 앱 등록 시 저장을 막고 오류 문구를 표시한다.

### 8.6 관리자 코드 인증
- 관리자는 관리자 버튼을 눌러 코드 입력 화면에 들어갈 수 있다.
- 올바른 코드를 입력하면 앱 추가 화면에 접근할 수 있다.
- 관리자 코드는 환경변수 또는 서버 측 검증 방식으로 관리하는 것을 권장한다.
- 단순히 프론트엔드 코드 안에 관리자 코드를 직접 적는 방식은 배포 후 노출될 수 있으므로 권장하지 않는다.

### 8.7 앱 추가
- 관리자는 새 앱의 제목, 링크, 카테고리, 태그, 썸네일, 설명을 입력할 수 있어야 한다.
- 필수 항목은 제목, 링크, 카테고리이다.
- 태그와 썸네일은 선택 항목으로 둘 수 있다.
- 썸네일이 없으면 기본 이미지를 표시한다.
- 저장 후 학생용 앱 목록에 새 앱이 반영되어야 한다.

### 8.8 썸네일 업로드
- 관리자는 이미지 파일을 업로드할 수 있어야 한다.
- 업로드 전 미리보기를 제공한다.
- 허용 형식은 jpg, jpeg, png, webp로 제한한다.
- 파일 크기는 2MB 이하를 권장한다.
- 이미지가 너무 큰 경우 업로드 전 압축하거나 리사이즈하는 것을 권장한다.
- 썸네일에는 대체 텍스트를 자동 또는 수동으로 제공한다.

### 8.9 앱 공개 상태
- 앱 등록 시 기본값은 공개로 설정한다.
- 비공개 앱은 학생 화면에 표시하지 않는다.
- 추후 관리자 화면에서 공개/비공개를 전환할 수 있다.

## 10. 저장 방식

### 10.1 1차 권장 방식
앱 추가와 썸네일 업로드가 필요하므로 정적 JSON만으로는 충분하지 않다.  
1차 구현에서는 다음 중 하나를 선택한다.

#### 선택지 A: Firebase 사용
- 앱 정보: Firestore
- 썸네일 이미지: Firebase Storage
- 관리자 코드 검증: Firebase Function 또는 서버리스 함수 권장

장점:
- 기존 Firebase 사용 경험과 연결 가능
- 앱 추가 후 바로 반영 가능
- 이미지 업로드 관리 가능

#### 선택지 B: Netlify Functions + 저장소
- 앱 정보: 외부 DB 또는 Netlify Blobs
- 썸네일 이미지: 별도 스토리지
- 관리자 코드 검증: Netlify Functions 환경변수

장점:
- Netlify 배포 흐름과 잘 맞음
- 관리자 코드를 프론트엔드에 노출하지 않을 수 있음

### 10.2 단순 임시 구현 방식
초기 테스트만 한다면 브라우저 localStorage에 앱 정보를 저장할 수 있다.  
하지만 이 방식은 같은 컴퓨터에서만 보이고, 다른 학생 기기에는 반영되지 않으므로 실제 운영용으로는 적합하지 않다.

## 11. UI 디자인 요구사항

### 11.1 전체 분위기
- 앱 런처처럼 한눈에 들어오는 카드형 UI
- 초등학생이 사용하기 쉽게 큰 카드와 큰 버튼 사용
- 썸네일 이미지를 활용해 앱 성격을 직관적으로 보여줌
- 색상은 카테고리별로 약하게 구분하되, 태그와 텍스트 정보도 함께 제공

### 11.2 추천 레이아웃

```text
┌────────────────────────────────────┐
│ 우리반 학습 포털                    │
│ 필요한 학습 앱을 골라 시작하세요.   │
│ [앱 이름이나 #태그 검색]             │
│                              [관리자]│
├────────────────────────────────────┤
│ [전체] [수업] [학습] [놀이] [학급운영]│
├────────────────────────────────────┤
│ 자주 쓰는 태그                       │
│ [#독서] [#퀴즈] [#경제] [#빙고]       │
├────────────────────────────────────┤
│ [앱 카드] [앱 카드] [앱 카드]        │
│ [앱 카드] [앱 카드] [앱 카드]        │
└────────────────────────────────────┘
```

### 11.3 앱 카드 디자인
- 썸네일은 카드 상단에 16:9 비율로 표시한다.
- 제목은 굵고 크게 표시한다.
- 설명은 최대 2줄로 제한한다.
- 카테고리는 작은 라벨로 표시한다.
- 태그는 칩 형태로 표시한다.
- 시작하기 버튼은 카드 하단에 고정한다.

## 13. 수용 기준

### 13.1 메인 화면
- 사용자가 포털에 접속하면 포털 이름, 검색창, 카테고리 필터, 태그 영역, 앱 카드 목록이 보여야 한다.

### 13.2 카테고리
- 카테고리는 전체, 수업, 학습, 놀이, 학급운영으로 표시되어야 한다.
- 과목별 카테고리는 사용하지 않는다.

### 13.3 태그
- 앱 카드에는 태그가 표시되어야 한다.
- 태그를 누르면 같은 태그가 달린 앱만 표시되어야 한다.
- 태그와 카테고리는 동시에 적용될 수 있어야 한다.

### 13.4 검색
- 검색어 입력 시 앱 제목, 설명, 태그 기준으로 앱이 필터링되어야 한다.
- `#`을 포함한 태그 검색도 가능해야 한다.

### 13.5 관리자 모드
- 관리자는 관리자 코드를 입력해 앱 추가 화면에 들어갈 수 있어야 한다.
- 잘못된 코드를 입력하면 오류 문구가 표시되어야 한다.

### 13.6 앱 추가
- 관리자는 제목, 링크, 카테고리, 태그, 썸네일, 설명을 입력해 앱을 추가할 수 있어야 한다.
- 앱 등록 후 새 앱이 학생용 목록에 반영되어야 한다.
- 필수값이 빠지거나 URL 형식이 잘못되면 저장되지 않아야 한다.

### 13.7 썸네일
- 관리자는 썸네일 이미지를 업로드할 수 있어야 한다.
- 업로드한 썸네일은 앱 카드에 표시되어야 한다.
- 썸네일이 없으면 기본 이미지가 표시되어야 한다.

### 13.8 반응형
- 데스크톱에서는 3~4열 카드 그리드를 사용한다.
- 태블릿에서는 2열 카드 그리드를 사용한다.
- 모바일에서는 1열 카드 목록을 사용한다.

### 15.2 추천 데이터 흐름
```text
앱 등록 폼 입력
→ 관리자 코드 검증
→ 썸네일 업로드
→ 앱 정보 저장
→ 앱 목록 다시 불러오기
→ 학생용 포털에 반영
```

### 15.3 보안 메모
관리자 코드를 프론트엔드 코드에 직접 넣으면 브라우저 개발자 도구나 배포 파일에서 노출될 수 있다.  
따라서 실제 운영에서는 관리자 코드 검증을 서버리스 함수, Firebase Function, Firestore Security Rules, 또는 Firebase Auth와 함께 처리하는 것이 좋다.

단, 교내 테스트용 초기 버전에서는 임시로 간단한 코드 검증을 사용할 수 있다. 이 경우 강한 보안 기능이 아니라 실수 방지용 잠금 장치로 봐야 한다.

## 16. 미정 사항

- 포털 최종 이름
- 관리자 코드 검증 방식
- Firebase를 사용할지 Netlify Functions를 사용할지
- 앱 썸네일 권장 크기
- 앱 수정/삭제 기능을 1차에 포함할지 여부
- 앱 공개/비공개 기능을 1차에 포함할지 여부
- 태그 추천 목록을 고정할지 자동 생성할지
- 앱 정렬 기준을 최신순으로 할지, 수동 정렬순으로 할지
